# 参考 rico-md 的后续路线

本文记录 `md2wechat` 参考 `rico-md` 前端设计后的后续开发路线。后续涉及前端体验、主题系统、复制兼容性和图片能力时，优先回到本文判断方向。

参考仓库本地位置：`/tmp/rico-md`

参考仓库地址：https://github.com/ricocc/rico-md/

## 问题本质

`md2wechat` 的核心不是再做一个通用 Markdown 编辑器，而是降低公众号发布的不确定性。

它要解决三件事：

1. 文章能长期、稳定地保存在本地真实文件里。
2. Markdown 能被渲染成公众号可接受的排版结构。
3. 复制到公众号后台后，样式和结构尽量不丢失、不变形。

因此，后续路线应保留当前项目的本地文章库和 Go 服务优势，吸收 `rico-md` 的工作台体验、主题组织、编辑效率和复制前处理能力。不要直接照搬 `rico-md` 的纯前端 localStorage / IndexedDB 文档模型。

## 当前项目定位

当前 `md2wechat` 已经具备这些基础：

- React + Vite 前端。
- Go 本地服务。
- 真实目录和 `.md` 文件组成的文章库。
- 文章树、新建、重命名、删除、自动保存。
- Markdown 渲染到公众号内联样式 HTML。
- Clipboard API 复制富文本。

这些是后续演进的根基，尤其是本地真实文件存储，不应被前端临时存储模型替代。

## rico-md 可借鉴点

`rico-md` 值得借鉴的是编辑器产品完整度：

- 顶部工具栏、左侧文档栏、编辑区、预览区、右侧主题/代码面板、底部状态栏。
- Markdown 工具按钮、快捷键、同步滚动、字数和阅读时间统计。
- 桌面/手机预览切换。
- 多套公众号主题，按风格分类，并支持推荐和收藏。
- 代码块主题与文章主题分离。
- 粘贴、拖拽、上传图片，压缩后持久化。
- 复制到公众号前处理图片、表格、代码块、引用、列表和纯文本 fallback。
- 导入 Markdown、导出 Markdown、导出 HTML。

但不建议直接照搬：

- 文档用 localStorage 管理。
- 图片只放 IndexedDB。
- Vue + CDN 的架构。
- 以单页静态工具为核心的状态模型。

## 总体方向

推荐路线是：先升级前端工作台，再扩展主题系统，然后补齐编辑效率，最后强化公众号复制和图片能力。

优先级判断标准：

1. 先保证复制到公众号稳定。
2. 再提升写作和预览效率。
3. 再增加主题、导出、图片等扩展能力。
4. 不为了视觉完整度牺牲本地文件存储和安全路径校验。

## 第 1 阶段：升级工作台壳层

目标：让当前界面从三栏基础工具升级成更完整的写作工作台。

建议内容：

- 保留左侧文章库、编辑器、预览三块核心结构。
- 顶部改为主操作区，放置品牌、保存状态、主题、代码、设置、复制按钮。
- 底部增加状态栏，展示保存状态、字数、词数、预计阅读时间、最后保存时间。
- 编辑区增加标题输入或当前文章名区域。
- 预览区增加手机/桌面模式切换。
- 加入右侧面板容器，为后续主题、代码、设置功能留位置。

实施原则：

- 先改 UI 结构，不改变后端 API。
- 不引入新运行时。
- 如果 `App.tsx` 继续膨胀，先拆组件再加复杂功能。

## 第 2 阶段：多主题系统

目标：从当前单主题扩展到可选择、可维护的多主题系统。

建议内容：

- 将 `web/src/core/themes.ts` 扩展为主题注册表。
- 增加 `ThemeId` 联合类型或集中注册结构。
- 支持主题分类、推荐标记和显示名。
- 前端增加主题面板，选择后实时更新预览。
- 当前选中主题持久化到 localStorage。
- 复制富文本时使用当前主题，而不是固定默认主题。

主题来源：

- 可以参考 `/tmp/rico-md/assets/styles/themes/` 的主题设计。
- 不建议直接保留 JS 模块结构，应该整理成当前项目的 TypeScript 类型。
- 第一批只迁移少量代表主题，先跑通结构，再批量补齐。

关键风险：

- 微信公众号对 CSS 支持有限，主题必须继续以内联样式为核心。
- 新主题不能破坏当前默认主题的复制稳定性。

## 第 3 阶段：编辑效率功能

目标：减少手写 Markdown 语法和频繁切换窗口的成本。

建议内容：

- 增加 Markdown 工具栏：H1、H2、H3、加粗、斜体、行内代码、链接、引用、有序列表、无序列表、代码块、表格、图片语法。
- 支持常用快捷键：`Ctrl/Cmd+S`、`Ctrl/Cmd+B`、`Ctrl/Cmd+I`、`Ctrl/Cmd+K`、`Tab` 缩进。
- 支持编辑器和预览同步滚动。
- 文章库支持搜索。
- 统计字符数、词数、预计阅读时间。

实施原则：

- 这些功能主要是前端状态和文本操作，不应改变文章文件格式。
- 工具栏行为要基于 textarea selection 实现，避免引入重型编辑器。
- 先覆盖常用语法，不追求一次做成完整 Markdown IDE。

## 第 4 阶段：强化复制到公众号兼容性

目标：让复制结果更稳定，这是项目核心价值。

建议内容：

- 复制前对表格做公众号兼容处理。
- 复制前对代码块做专门转换，避免代码样式和换行丢失。
- 复制前对引用块做颜色和背景归一化。
- 连续图片可以在复制时转换为表格布局，提高公众号后台兼容性。
- 生成更好的 `text/plain` fallback，而不是简单使用原 Markdown。
- 保持渲染预览和复制 HTML 的处理链路可区分：预览适合浏览器，复制适合公众号。

技术方向：

- 继续沿用当前 unified / rehype 渲染链。
- 不为了照搬 `rico-md` 而切换到 `markdown-it`。
- 只有在明确需要数学公式、CJK emphasis 补丁或 markdown-it 插件生态时，再评估替换渲染引擎。

## 第 5 阶段：图片能力

目标：支持公众号写作中常见的截图、拖拽和本地图片流程。

推荐方向：

- 不照搬 `rico-md` 的 IndexedDB 图片存储作为主方案。
- 当前项目应采用文件系统优先：图片保存到文章库内的 assets 目录。
- Markdown 中插入相对路径，保证文章和图片可迁移。
- Go API 负责图片写入、路径校验、读取和删除。
- 前端负责粘贴、拖拽、上传和可选 Canvas 压缩。
- 复制到公众号时，将本地图片转换为 Base64 或公众号更容易接受的结构。

建议后端能力：

- 上传图片接口。
- 读取图片资源接口。
- 删除未引用图片的清理能力可以后置，不作为第一版必需。

关键风险：

- 图片路径必须防止越界。
- 大图复制到剪贴板可能慢，需要有进度或错误提示。
- GIF 和 SVG 不应默认压缩，以免破坏动画或矢量内容。

## 第 6 阶段：发布级体验

目标：把工具从能用推进到适合长期日常写作。

建议内容：

- 导入 `.md` 文件。
- 导出 Markdown。
- 导出 HTML。
- 复制纯文本。
- 主题预设保存。
- 显示当前文章库路径。
- 支持打开文章库目录。
- 更明确的错误恢复提示。
- Windows 绿色包体验完善。

这些能力放在后期，是因为它们依赖前面的主题、复制、图片链路稳定。

## 最小可执行路线

如果要快速开始，建议先做三个闭环：

1. 工作台壳层升级。
2. 主题面板和多主题渲染。
3. 当前主题参与复制富文本。

这三步能最快把 `rico-md` 的前端设计价值迁移过来，同时不触碰图片存储和后端资产 API 这些高风险部分。

## 后续开发约束

- 保留当前真实文件文章库，不退回纯前端文档存储。
- 不主动替换渲染引擎。
- 不为了 UI 一次性引入大型组件库。
- 新功能优先围绕公众号复制稳定性和写作效率。
- 主题和复制处理要可测试、可回退。
- 改动前仍按项目规范先说明计划，确认后再动手。
